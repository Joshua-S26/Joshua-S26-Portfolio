Power BI portfolio project · Report as code

Olist delivery insights

A Power BI executive dashboard, authored as code

A single-page Power BI executive dashboard, plus a written case study, built on the Olist Brazilian e-commerce dataset: about 100,000 orders. It answers a real marketplace question: late deliveries lower review scores and put repeat revenue at risk, so where is it happening, why, and how much revenue is exposed.

FIG. 1 · Executive Overview page, captured in Power BI Desktop

The headline finding

Late deliveries put about $2.9M (roughly 21% of sales) of repeat revenue at risk, and on-time delivery is the fastest lever to protect it.

Money figures are in Brazilian Reais (R$), the dataset's currency. The dollar sign is used for readability; these are not US dollars.

What the project is

This is a personal analytics portfolio project, built on the public Olist Brazilian e-commerce dataset, not a client or employer deliverable. It pairs a single-page Power BI executive dashboard with a written case study. The dashboard is built as one decisive page: a KPI strip with targets, a delivery-versus-rating scatter, a state scorecard, and a dynamic key-insight line.

Authored as code

The report is authored as code in Power BI Desktop developer mode. The semantic model is written in TMDL (Tabular Model Definition Language), and the report is in PBIR, the enhanced report format. Every table, DAX measure, and visual is a plain-text file that can be reviewed and diffed like source code, rather than a binary. The Power BI project is named OlistDeliveryInsights, and the full model and report code sit in the report/ folder of the repository.

The data model

The model is a star schema with two fact tables at different grains, the clearest modeling signal to get right: order lines for revenue, and order headers for delivery and satisfaction. Both share conformed Date and Customer dimensions. There is no direct relationship between the two fact tables; the headline measure, Revenue at Risk, links them virtually with TREATAS, so there is no ambiguous physical relationship.

TableGrainNotes
Dim_DateOne row per calendar dayMarked as the model's date table; drives time intelligence (Year, Quarter, Month, Month Year)
Dim_CustomerOne row per customer_idConformed: filters both fact tables; Customer City and Customer State drive geography
Dim_ProductOne row per product_idCategory, translated to English; relates to order lines only
Dim_SellerOne row per seller_idSeller City and Seller State; relates to order lines only
Fact_OrderItemsOne row per order lineprice and freight_value; the revenue grain
Fact_OrdersOne row per orderOrder Status, review_score, delivery_days, is_late, Delivered Date, Estimated Date; the delivery and satisfaction grain

All relationships are single-direction, one to many. Both fact tables relate to Dim_Date on purchase_date and to Dim_Customer on customer_id. Fact_OrderItems additionally relates to Dim_Product on product_id and to Dim_Seller on seller_id.

Storage and source

Storage mode is Import. Currency is the Brazilian Real (R$). The source is nine Olist CSVs, loaded through Power Query (M) using a fnLoadCsv helper and a CsvFolder parameter, so the source folder is configurable rather than hard-coded.
